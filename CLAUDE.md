## 1. 定位

## 2. 项目概述

**Trans2Former** 是一个浏览器优先的多格式文档转换工具，支持 Markdown、PDF、DOCX、XLSX、HTML 等多种格式的相互转换。

- **定位**：本地优先、隐私安全的文档格式转换工具
- **核心价值**：浏览器内完成转换，无需上传文件到服务器
- **技术特点**：纯前端转换引擎 + 可选 OCR + 高保真输出

## 4. 模块化开发约束

Trans2Former 采用清晰的模块化架构：

```
Trans2Former/
├── public/
│   ├── core/           # 核心转换引擎（预算 460KB）
│   ├── formats/        # 格式读写器（预算 512KB）
│   ├── workers/        # Web Workers（预算 128KB）
│   ├── vendor/         # 第三方库（预算 96MB）
│   └── app.js          # 主应用入口
├── src/
│   └── web-server.js   # Express 服务器
└── scripts/            # 测试和构建脚本
```

### 模块职责

- **public/core**: 文档模型、转换管道、格式注册、OCR 管线、验证引擎
- **public/formats**: 各格式的 reader/writer（markdown, pdf, docx, xlsx 等）
- **public/workers**: 后台转换 worker，避免阻塞 UI
- **public/vendor**: PDF.js, Tesseract.js, ONNX Runtime 等第三方引擎

### 开发约束

1. **模块边界严格**：跨模块修改前必须说明影响面
2. **禁止无边界扩散**：不得顺手重构无关模块
3. **资源预算守护**：每个目录有严格的大小预算，超出需要说明理由
4. **依赖方向单一**：formats → core，workers → core + formats
5. **公共模块谨慎**：core 模块变更必须提高审核等级

## 6. 阶段代码审核与 Issue 报告（强制）

### 审核时机（必须执行）

**⚠️ 每个开发阶段结束后必须执行全面代码审核，不得跳过**：

- ✅ 完成一个功能模块
- ✅ 修复一批 Bug
- ✅ 重构一个子系统
- ✅ 准备发布前
- ✅ 提交 PR 前

**审核命令**：
```bash
# 使用 Claude Code 内置的代码审核技能
/code-review --effort medium

# 或针对安全问题进行专项审核
/security-review
```

### 审核维度

1. **正确性**：逻辑是否正确，边界条件是否处理
2. **健壮性**：错误处理是否完善，异常路径是否覆盖
3. **安全性**：是否有注入、XSS、路径遍历等风险
4. **性能**：是否有性能瓶颈，内存泄漏
5. **可维护性**：代码是否清晰，注释是否充分
6. **可测试性**：是否易于测试，测试覆盖是否充分
7. **用户体验**：交互是否流畅，错误提示是否友好

### Issue 报告（强制要求）

**⚠️ 代码审核发现的问题必须记录为 GitHub Issue，不得仅口头说明或在文档中记录**

### Issue 标签体系

**严重级别**：
- `P0`: 阻断性问题，立即修复（数据损坏、安全漏洞、核心功能失效）
- `P1`: 严重问题，本版本必须修复（重要功能缺陷、明显的性能问题）
- `P2`: 重要问题，计划修复（非关键路径问题、优化建议）
- `P3`: 优化建议，择机修复（代码风格、注释改进）

**问题类型**：
- `bug`: 功能缺陷
- `security`: 安全漏洞
- `performance`: 性能问题
- `robustness`: 健壮性问题
- `ux`: 用户体验
- `test`: 测试相关
- `docs`: 文档相关
- `refactor`: 重构需求

**模块标签**：
- `module:core`: 核心模块
- `module:formats`: 格式转换
- `module:workers`: Worker 相关
- `module:ui`: 用户界面
- `module:server`: 服务端
- `module:ocr`: OCR 相关
- `module:pdf`: PDF 处理

### Issue 创建要求（必须包含）

每个发现的问题必须创建 Issue，包含以下内容：

1. **标题**：简洁描述问题（50 字符内）
2. **描述**：详细说明问题现象、影响范围、根本原因
3. **复现步骤**：如何触发问题（步骤清晰、可操作）
4. **期望行为**：正确的行为应该是什么
5. **实际行为**：当前错误的行为
6. **代码位置**：文件路径、函数名、行号
7. **标签**：严重级别 + 类型 + 模块（至少 3 个标签）
8. **关联**：相关的 PR、Commit、其他 Issue

**Issue 创建命令**：
```bash
gh issue create --title "标题" --body "描述" --label "P1,bug,module:core"
```

## 7. 质量、测试与覆盖率红线

### 覆盖率要求（强制）

- **整体覆盖率**: ≥ 80% （当前 81.38%）
- **分支覆盖率**: ≥ 70% （当前 71.95%）
- **函数覆盖率**: ≥ 80% （当前 85.56%）
- **目标覆盖率**: ≥ 85% （Phase 1 完成后）

### 测试分层

1. **单元测试**：核心逻辑、工具函数、格式解析器
2. **集成测试**：模块协作、转换管道、Worker 通信
3. **端到端测试**：关键用户路径（上传→转换→下载）
4. **性能测试**：大文件转换、内存使用、响应时间

### P0 路径（必须测试）

- 文件上传 → 格式识别 → 转换 → 预览 → 下载
- Markdown ⇄ HTML ⇄ PDF 转换
- DOCX/XLSX 读取和结构保留
- 错误处理和用户反馈
- OCR 功能（可选功能，但启用后必须工作）

### 测试命令

```bash
npm test                  # 运行所有测试
npm run coverage          # 运行测试并生成覆盖率报告
node scripts/xxx-test.js  # 运行单个测试
```

### 覆盖率未达标时

- ❌ 不得宣称阶段完成
- ⚠️ 必须说明未达标项、风险、补救计划
- 📋 登记为 P1 Issue，阻塞发布

## 8. AI 开发节制与 UI 体验

### 开发节制原则

1. **不过度超前**：只实现当前 PRD 和 IMPLEMENTATION_PLAN 中的内容
2. **不堆叠功能**：一次只做一个功能模块，不贪多
3. **优化后置**：发现可选优化时，登记为 Issue，不混入当前阶段
4. **保持简单**：优先简单可行的方案，避免过度设计

### UI 体验原则

Trans2Former 是面向普通用户的工具，必须简单易用：

1. **现代清晰**：采用现代、清晰的设计语言
2. **符合习惯**：遵循用户常见的操作习惯
3. **即时反馈**：操作后立即给出反馈（进度、成功、失败）
4. **错误友好**：错误信息清晰易懂，提供解决建议
5. **性能感知**：大文件转换显示进度，避免无响应
6. **可访问性**：支持键盘导航，ARIA 标签，对比度达标

### 禁止的 UI 模式

- ❌ 复杂的多步向导（除非必需）
- ❌ 隐藏关键功能
- ❌ 无必要的动画（影响性能）
- ❌ 模糊的错误提示（"出错了" "失败"）
- ❌ 无进度指示的长时间操作

## 9. 版本规则

采用语义化版本 `MAJOR.MINOR.PATCH`：

- **MAJOR**：不兼容的 API 变更、数据格式变更
- **MINOR**：向下兼容的新功能
- **PATCH**：向下兼容的 Bug 修复

当前版本：`2.3.0`

### 版本更新时机

- 添加新格式支持 → MINOR
- 优化现有转换质量 → PATCH
- 重构内部实现（不影响 API）→ PATCH
- 改变输出格式结构 → MAJOR
