## 1. 定位说明

本文档是 **Trans2Former 项目的 AI 协作入口**，定义了 AI 辅助开发的工作流程、质量标准和执行红线。

**⚠️ 强制性**：所有 AI 代理（Claude Code 等）在执行任务前必须完整阅读本文档，并严格遵守所有约束和红线。

### 项目定位
- **产品**: 浏览器优先的多格式文档转换工具（PDF/Word/Excel/Markdown/HTML/图片等）
- **技术栈**: 纯前端架构（浏览器端 + Node.js 本地服务器），无后端依赖
- **核心能力**: PDF.js、Tesseract.js、ONNX Runtime、PaddleOCR、Puppeteer
- **质量红线**: 测试覆盖率 ≥ 80%，分支覆盖率 ≥ 70%
- **开发规范**: DevDocsKit 个人开发者规范体系

## 2. 项目概述

**Trans2Former** 是一个浏览器优先的多格式文档转换工具，支持 Markdown、PDF、DOCX、XLSX、HTML 等多种格式的相互转换。

- **定位**：本地优先、隐私安全的文档格式转换工具
- **核心价值**：浏览器内完成转换，无需上传文件到服务器
- **技术特点**：纯前端转换引擎 + 可选 OCR + 高保真输出

## 4. 模块化开发约束

Trans2Former 采用清晰的模块化架构：

```
Trans2Former/
├── public/
│   ├── core/           # 核心转换引擎（预算 460KB）
│   ├── formats/        # 格式读写器（预算 512KB）
│   ├── workers/        # Web Workers（预算 128KB）
│   ├── vendor/         # 第三方库（预算 96MB）
│   └── app.js          # 主应用入口
├── src/
│   └── web-server.js   # Express 服务器
└── scripts/            # 测试和构建脚本
```

### 模块职责

- **public/core**: 文档模型、转换管道、格式注册、OCR 管线、验证引擎
- **public/formats**: 各格式的 reader/writer（markdown, pdf, docx, xlsx 等）
- **public/workers**: 后台转换 worker，避免阻塞 UI
- **public/vendor**: PDF.js, Tesseract.js, ONNX Runtime 等第三方引擎

### 开发约束

1. **模块边界严格**：跨模块修改前必须说明影响面
2. **禁止无边界扩散**：不得顺手重构无关模块
3. **资源预算守护**：每个目录有严格的大小预算，超出需要说明理由
4. **依赖方向单一**：formats → core，workers → core + formats
5. **公共模块谨慎**：core 模块变更必须提高审核等级

## 6. 阶段代码审核与 Issue 报告（强制）

### 审核时机（必须执行）

**⚠️ 每个开发阶段结束后必须执行全面代码审核，不得跳过**：

- ✅ 完成一个功能模块
- ✅ 修复一批 Bug
- ✅ 重构一个子系统
- ✅ 准备发布前
- ✅ 提交 PR 前

**审核命令**：
```bash
# 使用 Claude Code 内置的代码审核技能
/code-review --effort medium

# 或针对安全问题进行专项审核
/security-review
```

### 审核维度

1. **正确性**：逻辑是否正确，边界条件是否处理
2. **健壮性**：错误处理是否完善，异常路径是否覆盖
3. **安全性**：是否有注入、XSS、路径遍历等风险
4. **性能**：是否有性能瓶颈，内存泄漏
5. **可维护性**：代码是否清晰，注释是否充分
6. **可测试性**：是否易于测试，测试覆盖是否充分
7. **用户体验**：交互是否流畅，错误提示是否友好

### Issue 报告（强制要求）

**⚠️ 代码审核发现的问题必须记录为 GitHub Issue，不得仅口头说明或在文档中记录**

### Issue 标签体系

**严重级别**：
- `P0`: 阻断性问题，立即修复（数据损坏、安全漏洞、核心功能失效）
- `P1`: 严重问题，本版本必须修复（重要功能缺陷、明显的性能问题）
- `P2`: 重要问题，计划修复（非关键路径问题、优化建议）
- `P3`: 优化建议，择机修复（代码风格、注释改进）

**问题类型**：
- `bug`: 功能缺陷
- `security`: 安全漏洞
- `performance`: 性能问题
- `robustness`: 健壮性问题
- `ux`: 用户体验
- `test`: 测试相关
- `docs`: 文档相关
- `refactor`: 重构需求

**模块标签**：
- `module:core`: 核心模块
- `module:formats`: 格式转换
- `module:workers`: Worker 相关
- `module:ui`: 用户界面
- `module:server`: 服务端
- `module:ocr`: OCR 相关
- `module:pdf`: PDF 处理

### Issue 创建要求（必须包含）

每个发现的问题必须创建 Issue，包含以下内容：

1. **标题**：简洁描述问题（50 字符内）
2. **描述**：详细说明问题现象、影响范围、根本原因
3. **复现步骤**：如何触发问题（步骤清晰、可操作）
4. **期望行为**：正确的行为应该是什么
5. **实际行为**：当前错误的行为
6. **代码位置**：文件路径、函数名、行号
7. **标签**：严重级别 + 类型 + 模块（至少 3 个标签）
8. **关联**：相关的 PR、Commit、其他 Issue

**Issue 创建命令**：
```bash
gh issue create --title "标题" --body "描述" --label "P1,bug,module:core"
```

## 7. 质量、测试与覆盖率红线（强制）

### 覆盖率要求（不可突破的红线）

**⚠️ 以下是强制要求，未达标不得宣称任务完成**：

- **整体覆盖率**: ≥ 80% （当前 81.38%，必须保持）
- **分支覆盖率**: ≥ 70% （当前 71.95%，持续改进目标 ≥ 75%）
- **函数覆盖率**: ≥ 80% （当前 85.56%）
- **目标覆盖率**: ≥ 85% （Phase 1 完成后的目标）

**新增代码的覆盖率要求**：
- 所有新增功能必须有对应测试
- 新增代码的覆盖率不得低于 85%
- 核心模块（`public/core/`）新增代码覆盖率不得低于 90%

### 测试分层

1. **单元测试**：核心逻辑、工具函数、格式解析器
   - 测试纯函数、数据转换、算法实现
   - 每个工具函数至少 1 个测试用例
   
2. **集成测试**：模块协作、转换管道、Worker 通信
   - 测试模块间交互、数据流转
   - 每个转换路径至少 3 个测试用例（成功、失败、边界）
   
3. **端到端测试**：关键用户路径（上传→转换→下载）
   - 测试完整的用户操作流程
   - P0 路径必须有端到端测试覆盖
   
4. **性能测试**：大文件转换、内存使用、响应时间
   - 资源预算测试（每个目录有大小限制）
   - 内存泄漏检测
   - 响应时间基准

### P0 路径（必须测试，不得缺失）

- ✅ 文件上传 → 格式识别 → 转换 → 预览 → 下载
- ✅ Markdown ⇄ HTML ⇄ PDF 转换
- ✅ DOCX/XLSX 读取和结构保留
- ✅ 错误处理和用户反馈
- ✅ OCR 功能（可选功能，但启用后必须工作）

### 测试命令

```bash
# 运行所有测试
npm test

# 运行测试并生成覆盖率报告
npm run coverage

# 运行单个测试
node scripts/xxx-test.js

# 查看覆盖率报告
open coverage/index.html  # macOS
start coverage/index.html # Windows
```

### 覆盖率未达标时的处理

- ❌ **不得宣称阶段完成**
- ❌ **不得提交 PR**
- ⚠️ **必须说明未达标项、风险、补救计划**
- 📋 **登记为 P1 Issue，阻塞发布**

### 测试编写要求

1. **测试文件命名**：`scripts/功能名-test.js`
2. **测试结构**：清晰的 Arrange-Act-Assert 模式
3. **测试说明**：每个测试用例有清晰的注释说明测试目的
4. **断言清晰**：使用有意义的断言消息
5. **独立性**：测试之间相互独立，不依赖执行顺序
6. **可重复**：测试结果可重复，不依赖外部状态

## 8. AI 开发节制与 UI 体验

### 开发节制原则

1. **不过度超前**：只实现当前 PRD 和 IMPLEMENTATION_PLAN 中的内容
2. **不堆叠功能**：一次只做一个功能模块，不贪多
3. **优化后置**：发现可选优化时，登记为 Issue，不混入当前阶段
4. **保持简单**：优先简单可行的方案，避免过度设计

### UI 体验原则

Trans2Former 是面向普通用户的工具，必须简单易用：

1. **现代清晰**：采用现代、清晰的设计语言
2. **符合习惯**：遵循用户常见的操作习惯
3. **即时反馈**：操作后立即给出反馈（进度、成功、失败）
4. **错误友好**：错误信息清晰易懂，提供解决建议
5. **性能感知**：大文件转换显示进度，避免无响应
6. **可访问性**：支持键盘导航，ARIA 标签，对比度达标

### 禁止的 UI 模式

- ❌ 复杂的多步向导（除非必需）
- ❌ 隐藏关键功能
- ❌ 无必要的动画（影响性能）
- ❌ 模糊的错误提示（"出错了" "失败"）
- ❌ 无进度指示的长时间操作

## 9. 版本规则

采用语义化版本 `MAJOR.MINOR.PATCH`：

- **MAJOR**：不兼容的 API 变更、数据格式变更
- **MINOR**：向下兼容的新功能
- **PATCH**：向下兼容的 Bug 修复

当前版本：`2.3.0`

### 版本更新时机

- 添加新格式支持 → MINOR
- 优化现有转换质量 → PATCH
- 重构内部实现（不影响 API）→ PATCH
- 改变输出格式结构 → MAJOR

## 11. 执行红线（不可突破）

**⚠️ 以下行为严格禁止，违反将导致任务重做**：

1. ❌ **不覆盖用户未要求修改的内容**
   - 不得顺手重构无关模块
   - 不得添加未在计划中的功能
   - 不得修改不相关的代码格式

2. ❌ **不删除未确认用途的文件、配置、脚本**
   - 删除前必须确认文件的用途和依赖关系
   - 必须通过 `git log` 检查文件历史
   - 必须搜索代码中的引用

3. ❌ **不把未验证的猜测写成完成结论**
   - 所有修改必须经过测试验证
   - 不确定的问题必须标注为"待确认"
   - 必须区分"已验证"和"理论上应该"

4. ❌ **不在代码或文档中记录真实密钥、令牌、密码**
   - 使用占位符（如 `YOUR_API_KEY`）
   - 敏感信息必须通过环境变量配置
   - 提交前检查 `git diff` 是否包含敏感信息

5. ❌ **不引入未记录版本和用途的新依赖**
   - 添加依赖前必须说明理由和版本选择
   - 必须在文档中记录新依赖的用途
   - 优先使用项目现有依赖

6. ❌ **不跳过阶段代码审核、Issue 报告、覆盖率门禁**
   - 每个阶段结束必须执行代码审核
   - 审核发现的问题必须创建 Issue
   - 覆盖率未达标不得宣称完成

7. ❌ **不在测试未通过时宣称任务完成**
   - 所有测试必须通过
   - 覆盖率必须达标
   - 必须运行完整测试套件

8. ❌ **不修改资源预算而不说明理由**
   - 资源预算是经过设计的约束
   - 超出预算必须说明原因和影响
   - 必须评估对性能的影响

### 违反红线的处理

- 🔴 **立即停止任务**
- 🔴 **回滚所有修改**
- 🔴 **重新制定计划**
