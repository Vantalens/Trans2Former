# Trans2Former 项目反馈与修复行动清单

日期：2026-06-21  
对象：项目本人复盘  
状态：需要立即处理

## 一、结论摘要

Trans2Former 的产品方向清晰，已经建立较完整的多格式转换架构、自动化测试、质量报告、安全边界和项目文档。但根据现阶段实际测试反馈，当前版本的所有格式转换功能均完全失效，核心使用流程无法完成。

因此，项目当前不能按“早期可用产品”评价，更准确的状态是：

> 工程基础较完整，但核心功能存在系统性失效，当前版本不可用。

在转换功能恢复并通过真实端到端验证前，应暂停新增格式、界面扩展和非必要优化，将全部开发资源集中到核心转换链修复。

## 二、关键问题

### P0：所有格式转换功能完全失效

现阶段实际测试发现，所有格式转换均无法正常完成。这是项目的最高优先级阻断问题，直接影响：

- 全部输入与输出格式；
- 转换、预览、下载和质量报告主流程；
- 项目对外宣称的核心能力；
- 当前版本的可用性与发布资格。

目前尚未完成对故障触发条件和根因的独立复现，因此不能提前认定是格式解析器、路由器、Worker、UI 状态管理或输出下载中的某一层导致。修复工作必须先建立稳定复现，再根据证据定位公共故障点。

## 三、自动化测试与实际结果的矛盾

本次检查中，`npm test` 以及独立覆盖率测试均通过，覆盖率结果为：

- 行/语句覆盖率：81.28%；
- 分支覆盖率：71.94%；
- 函数覆盖率：85.54%。

但是，实际使用测试显示所有格式转换均失效。这说明现有自动化测试存在重要盲区：测试通过只能证明被测试的模块、契约和样例符合预期，不能证明真实用户路径可用。

可能存在的测试缺口包括：

- 部分“浏览器测试”主要验证静态服务、源码结构或函数存在性，没有驱动真实浏览器完成上传、转换、预览和下载；
- 格式测试大量直接调用底层转换函数，可能绕过 UI、Worker、文件读取和状态管理链路；
- 缺少覆盖全部格式的真实文件端到端测试；
- 缺少针对最终可下载文件内容的独立解析和校验；
- CI 仅覆盖 Ubuntu 上的 Node 测试，没有覆盖真实 Tauri 桌面运行环境。

因此，现有测试结果不能继续作为“全部格式正常工作”或“版本具备发布条件”的充分证据。

## 五、修复行动清单

### 第一阶段：建立可靠复现

1. 记录运行入口、操作系统、Node/Tauri 版本和启动方式。
2. 选择最简单的 `TXT → TXT` 或 `Markdown → HTML` 路径作为最小复现。
3. 保存浏览器控制台、服务端日志、Worker 错误和网络请求结果。
4. 确认失败发生在文件读取、转换路由、Worker 执行、结果回传、预览还是下载阶段。
5. 对 Web 入口和 Tauri 入口分别复现，判断是否为共享核心故障。

### 第二阶段：定位公共故障点

优先检查以下共享链路：

1. 文件内容读取及二进制/文本载荷规范化；
2. 输入格式检测与输出格式路由；
3. 主线程与 Web Worker 之间的消息协议；
4. `DocumentModel` 创建、校验及转换模型映射；
5. 转换结果回传、状态更新和错误处理；
6. Blob、Object URL、文件名和下载动作生成；
7. 浏览器 CSP、模块加载和 vendor 资源路径；
8. Tauri 环境与普通浏览器环境之间的差异。

如果所有格式同时失效，应先寻找共享入口或公共基础设施问题，不应逐个格式解析器分散修复。

### 第三阶段：实施修复

1. 为最小复现路径增加一个能够稳定失败的回归测试。
2. 修复公共根因，不以隐藏错误、强制降级或跳过校验作为完成标准。
3. 确保错误信息包含阶段、格式、错误代码和安全处理后的诊断信息。
4. 最小路径恢复后，再逐步扩展到文本、结构化数据、OOXML、PDF、图片/OCR等格式组。

### 第四阶段：全格式真实验证

每一种公开支持的输入格式至少使用一个真实样例，验证：

- 文件能够正确读取；
- 转换任务能够完成；
- 预览内容与输出一致；
- 下载文件能够被对应软件或独立解析器重新打开；
- 中文、特殊字符和基本结构未丢失；
- 发生降级时向用户明确展示，不得静默成功；
- Web 与桌面入口结果一致。

验证结果应形成格式矩阵，逐项记录通过、失败、降级和暂不支持，不能以单一总通过率替代。

### 第五阶段：补齐测试体系

1. 引入 Playwright 等真实浏览器端到端测试。
2. 覆盖“上传 → 选择格式 → 转换 → 预览 → 下载”的完整用户流程。
3. 对下载结果进行二次解析，验证文件不是空文件、错误页或无效容器。
4. 为每种公开格式保留稳定的最小真实样例。
5. 在 CI 中增加至少一个真实浏览器任务。
6. 增加 Windows Tauri 构建及桌面 smoke 测试。
7. 将端到端测试设置为发布阻断门禁。

## 六、修复完成标准

只有同时满足以下条件，才能认定本次问题已修复：

- 已明确记录并解释根因；
- 最小复现用例在修复前失败、修复后通过；
- 所有公开支持格式均完成真实文件转换验证；
- Web 与 Tauri 主流程均可完成转换和下载；
- 输出文件可由独立工具重新打开或解析；
- 自动化端到端测试已覆盖核心用户流程；
- 原有单元、集成、覆盖率、资源预算和安全测试继续通过；
- 不存在将失败错误地显示为成功的情况；
- 未通过的格式从产品能力矩阵和界面中暂时移除或明确标记为不可用。

## 七、现阶段开发优先级

建议按以下优先级执行：

1. **P0：恢复所有格式共用的核心转换链。**
2. **P0：建立真实浏览器端到端回归测试。**
3. **P0：逐项验证公开格式并修正能力矩阵。**
4. **P1：修复 `Cargo.lock` 漂移，恢复可复现桌面构建。**
5. **P1：增加 Windows/Tauri CI 和真实安装验证。**
6. **P2：拆分 `public/app.js` 和大型格式模块。**
7. **P3：核心稳定后再考虑新增格式、界面功能和性能优化。**

## 八、最终反馈

Trans2Former 已经积累了有价值的架构、格式模型、质量机制和自动化测试，但项目现阶段最严重的问题不是功能数量不足，而是自动化指标与真实可用性脱节。

下一阶段应以“真实用户能够完成转换并获得有效文件”为唯一核心目标。所有修复和测试工作都应围绕这一目标展开。在全格式端到端验证完成以前，不应继续将当前版本描述为可用或具备发布条件。
