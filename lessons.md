## 1. 技术决策

### 1.1 浏览器优先架构

**决策**：采用浏览器内转换，不依赖服务器端处理

**理由**：
- 隐私保护：用户文件不离开本地
- 降低成本：无需服务器算力
- 离线可用：无网络也能使用
- 响应快速：无网络延迟

**结果**：✅ 成功
- 用户反馈积极
- 部署成本极低
- 适合个人和小团队使用

**教训**：
- 需要处理浏览器内存限制（大文件分块）
- Web Workers 必不可少（避免 UI 冻结）
- 需要渐进式加载第三方库（vendor 按需加载）

### 1.2 DocumentModel 中间表示

**决策**：设计统一的 DocumentModel 作为所有格式的中间表示

**理由**：
- 解耦输入和输出格式
- 便于添加新格式支持
- 统一的质量验证
- 便于实现批量操作

**结果**：✅ 成功
- 新增格式只需实现 reader 或 writer
- 转换质量可控
- 便于测试和验证

**教训**：
- 模型设计要足够灵活（支持表格、图像、固定布局）
- 需要明确的降级策略（不是所有元素都能完美转换）
- 元数据和质量报告很重要

### 1.3 OCR 可选而非默认

**决策**：OCR 功能按需加载，不默认启用

**理由**：
- 模型文件大（20MB+）
- 性能开销高
- 大部分文档不需要 OCR

**结果**：✅ 成功
- 首次加载快速
- 内存占用低
- 需要时才下载模型

**教训**：
- 需要清晰的 UI 提示（何时需要 OCR）
- 模型下载进度必须可见
- 需要降级方案（OCR 失败时）

### 1.4 三层验证管道

**决策**：SSIM（图像相似度）+ OCR回读 + 规则差异 三层验证

**理由**：
- 单一验证维度不够
- 不同格式需要不同验证方式
- 用户需要知道转换质量

**结果**：✅ 成功
- 质量问题可追踪
- 降级建议更准确
- 用户信任度提高

**教训**：
- 验证层级要异步（避免阻塞）
- 验证成本要可控（大文件跳过图像验证）
- 验证结果要可解释（用户能理解）

## 2. 遇到的主要问题

### 2.1 PDF 文本提取空白

**问题**：生成的 PDF 在阅读器中无法选择和复制文本

**原因**：
- CID 字体缺少 ToUnicode CMap
- PDF 阅读器无法将字符代码映射到 Unicode

**解决方案**：
- 添加 buildToUnicodeCMap() 生成标准 ToUnicode CMap
- 在字体字典中添加 /ToUnicode 引用
- 使用 Identity-H 映射（适用于 BMP 字符）

**状态**：✅ 已修复（v2.3.0）

**遗留问题**：
- 代理对支持（CJK Extension B、Emoji）需要 4 字节 codespace

### 2.2 大文件主线程冻结

**问题**：转换大文件时 UI 无响应

**原因**：
- shouldUseLargeTextPreview API 签名不匹配
- 大文本降级策略未生效
- 渲染在主线程同步执行

**解决方案**：
- 修正 API 调用为对象参数
- 实现渐进式预览
- 使用虚拟滚动列表
- 关键操作移到 Web Worker

**状态**：✅ 已修复（v2.3.0）

### 2.3 资源预算持续增长

**问题**：public/core 目录大小从 322KB 增长到 456KB

**原因**：
- OCR 管线功能扩展
- PDF ToUnicode 支持
- 验证引擎完善

**解决方案**：
- 调整预算阈值到 460KB
- 在注释中记录增长理由
- 定期审查代码体积

**教训**：
- 预算要有余量（不要卡死在边界）
- 功能扩展前评估体积影响
- 考虑代码分割和按需加载

### 2.4 测试覆盖率提升困难

**问题**：分支覆盖率停滞在 70% 左右

**原因**：
- 错误路径测试不足
- 边界条件未覆盖
- 异步代码测试复杂

**解决方案**：
- 针对性添加错误路径测试
- 使用 mock 简化异步测试
- 优先覆盖核心模块

**状态**：🔄 进行中
- 当前 71.95%，目标 80%+

## 3. 性能优化经验

### 3.1 分块处理

**场景**：大文本文档（10MB+）

**优化**：
- 按段落分块
- 增量渲染
- 虚拟滚动

**效果**：
- 10MB 文本转换从 5000ms 降到 1500ms
- 内存占用降低 60%

### 3.2 Transferable 对象

**场景**：Worker 传输大量数据

**优化**：
- 使用 Transferable 转移 ArrayBuffer 所有权
- 避免数据复制

**效果**：
- Worker 通信开销降低 80%

### 3.3 Web Workers

**场景**：CPU 密集型转换

**优化**：
- 核心转换逻辑移到 Worker
- 主线程只负责 UI 更新

**效果**：
- UI 响应性显著提升
- 可以同时处理多个文件

## 4. 安全问题处理

### 4.1 ZIP Bomb 防护

**风险**：恶意 ZIP 文件（高压缩比，解压后巨大）

**防护措施**：
- 限制解压后大小
- 检查压缩比
- 小文件豁免（<64 bytes）

**状态**：✅ 已实现

### 4.2 路径遍历

**风险**：ZIP 中的文件路径包含 `../`

**防护措施**：
- 规范化路径
- 拒绝包含 `..` 的路径
- 限制在安全目录内

**状态**：✅ 已实现

### 4.3 XSS 防护

**风险**：HTML 输出包含用户输入的脚本

**防护措施**：
- HTML 实体转义
- CSP 头部
- 不信任 HTML 输入的脚本标签

**状态**：✅ 已实现

## 6. 代码质量

### 6.1 模块化

**原则**：
- 单一职责
- 清晰边界
- 依赖注入

**实践**：
- core 只包含纯逻辑
- formats 独立可测
- workers 隔离副作用

### 6.2 错误处理

**原则**：
- 明确的错误类型
- 上下文信息充足
- 用户友好的提示

**实践**：
- ConversionError 统一错误
- 错误码和分类
- 降级建议

### 6.3 注释和文档

**原则**：
- 代码即文档（清晰命名）
- 注释说明"为什么"，不是"做什么"
- 关键算法详细说明

**实践**：
- JSDoc 类型标注
- Issue 引用
- 算法来源引用

## 8. 避坑指南

### 8.1 不要...

- ❌ 在主线程做重计算（使用 Worker）
- ❌ 同步读取大文件（使用流式处理）
- ❌ 忽略错误路径（错误处理和测试同样重要）
- ❌ 过度优化（先工作，再优化，最后证明）
- ❌ 引入大依赖（vendor 按需加载）
- ❌ 假设用户输入安全（始终验证和转义）
- ❌ 硬编码配置（使用配置文件）
- ❌ 跳过测试（覆盖率是质量保证）

### 8.2 记住...

- ✅ 小步提交（原子化、可回滚）
- ✅ 测试先行（TDD when possible）
- ✅ 文档同步（代码和文档一起更新）
- ✅ 用户体验优先（性能、可访问性、错误提示）
- ✅ 安全第一（输入验证、输出转义、资源限制）
- ✅ 渐进增强（核心功能先行，高级功能可选）
- ✅ 定期审核（代码、依赖、安全）
- ✅ 保持简单（KISS principle）
