# Product Strategy

版本：v0.3.0
状态：生效
最后更新：2026-05-03

## 产品定位

Trans2Former Desktop 是专业级、本地优先、零上传、多格式、高质量的桌面端格式转换处理器。

它不是普通在线转换网站，也不是单纯 AI/RAG 工具。最终产品形态是：

```text
Tauri 桌面壳
+ Web-GUI 前端
+ TypeScript 转换核心
+ Web Worker / WASM
+ 本地插件系统
```

当前仓库仍保留浏览器 Web 应用作为转换核心和界面验证底座；后续桌面版本必须复用这套 Web-GUI 和本地转换核心，而不是引入 Electron、远程转换 API 或传统原生 GUI 重写。

## 不变原则

- 用户数据绝对安全：文档处理阶段本地执行，不上传、不遥测、不留存文档内容。
- 零云端处理：不提供云端文档处理、远程转换、远程 OCR、远程转写或远程 AI 增强。
- 桌面 Web-GUI：Tauri 承载 Web-GUI，Web 技术负责编辑、预览、质量报告和版本视图。
- 本地转换核心：不依赖 Microsoft Office、LibreOffice、Pandoc、Electron、Playwright 或云端转换服务。
- 中间模型优先：新增格式必须走 `input -> DocumentModel -> output`，避免 `N * N` 转换路线。
- 模块插件优先：核心包保持小而可用，热门基础格式免下载，重格式和可选能力按用户需求下载或加载。
- 高保真攻坚：难格式不是回避项，OFD、PDF、Office 复杂文档和本地 OCR 是长期差异化攻坚方向。
- 可验证交付：每个阶段都必须有样例、自动化测试、质量报告和可解释 warnings。

## 产品壁垒

- 桌面专业工作台：文件队列、转换配置、输入预览、结构视图、输出编辑、质量报告、版本历史和插件管理在同一工作台完成。
- 动态编辑：用户可以编辑标准化后的文档结构、块内容、表格、图片引用和输出参数。
- 实时预览：输入编辑、格式选择、转换 warnings、质量报告和输出预览尽量实时反馈。
- 上传文件大小无限制目标：不设置人为固定 MB/GB 上限；实际能力由用户设备资源决定。
- 动态分块不降质：单个超大文件可按语义子模块转换后合并，结果必须与直接转换语义等价。
- 行业顶尖质量：格式适配器必须有样例、快照、降级说明、性能基准和人工可读质量准则。
- OFD 高保真：OFD 是政务、公文、票据场景的战略攻坚格式，不以“能打开”为目标，而以本地高保真和可解释质量报告为目标。
- 热门格式免下载：基础包必须覆盖 Markdown、HTML、TXT、JSON、CSV、XML、常见图片、PDF text、DOCX/XLSX/PPTX/EPUB input 等高频路径。
- 按需能力扩展：用户只为当前任务加载需要的重格式、高保真渲染器、本地模型或 OFD 插件，减少默认资源占用。

## 体验目标

- 体验像成熟现代 PC 桌面软件，而不是网页 demo：主界面服务操作流，不堆说明文案。
- 丝滑但低占用：默认不扫描和加载重插件，大任务走 Worker、Transferable、虚拟滚动和渐进预览。
- 安全但不打扰：local-only、no-upload、插件权限和历史保存状态可见，但不反复弹窗打断用户。
- 专业但可理解：质量报告、warnings、diff 和版本历史必须让用户知道哪里降级、为什么降级、如何修复。
- 可恢复：取消、失败、插件崩溃和预览异常不能丢失用户编辑，也不能保留可误下载的旧结果。

## 市场切入

- 第一阶段服务专业用户对“本地、安全、多格式、可解释”的转换需求，而不是普通在线转换站。
- 基础盘优先覆盖 `Office/PDF/HTML/Data -> Markdown/HTML/JSON` 和常见输出。
- AI/RAG 友好 Markdown 是重要场景，但不是产品唯一定位。
- 高保真 PDF/OFD/Office 转换是长期壁垒，必须分阶段建立样例库、质量指标和插件化能力。
- 隐私、本地转换、多文件批量、可解释降级、动态编辑、实时预览和本地版本控制是桌面产品的核心差异化。

## 当前阶段策略

P0/P1/P2/P3 的核心链路已经完成，后续不应继续用“增加更多格式数量”作为主要进度指标。当前阶段更重要的是：

- 架构收敛：拆分工作台大文件，降低后续功能互相影响的风险。
- 质量证据：每个重格式必须有公开样例、快照、质量报告和降级说明。
- 真实插件加载：把插件从策略和 GUI 管理推进到 Worker/WASM 沙箱执行。
- 高保真攻坚：OFD、本地 OCR/layout/table 和高保真 Office/PDF 输出按插件化、样例驱动推进。
- 桌面发布：明确 Web preview 包和真正 Tauri 安装包的边界。

详细调研见 [MARKET_RESEARCH_2026-04-26.md](MARKET_RESEARCH_2026-04-26.md)。

## 数据安全底线

- 不上传文档、图片、转换结果、错误详情、文件名、文档片段或用户编辑内容。
- 不接入第三方转换 API、云端 OCR、云端转写、云端 AI、分析 SDK 或遥测 SDK。
- 不提供在线网页抓取、YouTube extraction、远程日志上传或自动错误报告上传。
- 错误详情面板必须区分“可复制诊断信息”和“可能包含用户内容的原始片段”。
- IndexedDB/localStorage/桌面本地存储只能保存用户明确允许的历史、偏好和临时缓存，并提供清除入口。
- 插件安装可以联网下载代码和资源，但不得接触用户文档；文档处理、编辑、预览和导出阶段必须禁联网。

安全细则见 [SECURITY_POLICY.md](SECURITY_POLICY.md)。

## 资源策略

- 默认包只包含 `core + format-basic + desktop shell` 的必要能力。
- `format-basic` 必须覆盖高频轻量格式，保证常见转换无需下载即可使用。
- 基础热门格式可以内置；高保真增强、OFD、本地 OCR、本地版面分析、本地表格恢复模型必须插件化或按需加载。
- 插件分发默认采用 GitHub Releases，应用内提供下载板块和更新板块，不自建插件分发后端。
- 模块插件必须通过 manifest 声明格式能力、体积预算、依赖、安全模式、加载方式和失败降级路径。
- 插件下载必须由用户需求触发，不能因为支持超广格式而默认下载所有能力。
- 重格式如果要晋升为基础内置能力，必须通过资源预算、安全、本地处理和质量基准评审。
- `npm test` 必须包含资源预算和本地安全检查，防止核心包无意引入重依赖、远程 SDK、网络处理路径或过大资产。
