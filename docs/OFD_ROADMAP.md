## 定位

OFD（Open Fixed-layout Document，开放版式文档）是 Trans2Former 的**战略攻坚格式**，定位为核心本地能力，不是边缘研究项。

## 当前状态

### 实现级别：L0（容器级基础解析）

**已实现**：
- ✅ OFD 容器解析（ZIP 结构读取）
- ✅ 基础元信息提取（Document.xml, PublicRes.xml）
- ✅ 文件结构验证

**当前限制**：
- ❌ 正文内容不完整提取
- ❌ 版式信息未解析（页面、图层、坐标）
- ❌ 字体、图片、矢量图形未处理
- ❌ 签章、水印、书签不支持

**输入路径**：
- 状态：实验性输入
- 质量：L0 级（仅容器信息）
- 警告：强制 `PATH_NOT_RECOMMENDED`

**输出路径**：
- 状态：不支持
- 原因：高保真输出需要完整版式引擎

## 攻坚路线

### L0 → L1：内容提取（当前阶段后续）

**目标**：完整提取文本内容，保留基本结构

**工作项**：
1. 解析 Content.xml（页面内容）
2. 提取 TextObject（文本对象）
3. 处理 PathObject（路径对象）基础信息
4. 提取 ImageObject（图片对象）引用

**输出**：
- OFD → Markdown（纯文本，丢失版式）
- OFD → HTML（语义内容，自适应布局）
- OFD → TXT（纯文本）

**时间**：2-3 周

### L1 → L2：版式保留

**目标**：保留页面版式信息，支持固定布局输出

**工作项**：
1. 解析页面尺寸和坐标系
2. 处理图层和 Z-order
3. 支持绝对定位文本块
4. 提取字体信息（需要 font fallback）
5. 处理图片嵌入和引用

**输出**：
- OFD → PDF（固定版式，保留布局）
- OFD → HTML（absolute positioning）

**时间**：4-5 周

### L2 → L3：高保真渲染

**目标**：接近原始 OFD 的视觉效果

**工作项**：
1. 矢量图形渲染（PathObject 完整支持）
2. 复杂样式（渐变、透明度、混合模式）
3. 字体嵌入和渲染
4. 表格识别和结构化
5. 签章和水印处理

**输出**：
- OFD → PDF（高保真）
- OFD → PNG/JPEG（栅格化输出）

**时间**：6-8 周

### L3 → L4：OFD 输出

**目标**：支持生成标准 OFD 文件

**工作项**：
1. OFD writer 实现（OFD.xml, Document.xml, Content.xml）
2. 资源打包（字体、图片、签章）
3. 版式计算和页面生成
4. 签章和加密支持
5. 验证和质量检查

**输出**：
- Markdown → OFD
- HTML → OFD
- PDF → OFD（固定版式互转）

**时间**：8-10 周

## 技术架构

### 核心本地模块

**位置**：`public/formats/ofd.js`（核心内置）

**依赖**：
- ZIP 容器解析：`public/core/zip-container.js`
- XML 解析：浏览器原生 DOMParser
- 字体处理：FontFace API（浏览器原生）
- 矢量渲染：Canvas 2D / SVG

**不依赖**：
- ❌ 云端 API
- ❌ LibreOffice / Pandoc
- ❌ 外部 OFD SDK

### 模型资源

**字体包**（如需）：
- 通过 vendor 脚本下载到 `public/vendor/fonts/`
- SHA-256 校验
- 不入 git

**测试样例**：
- `samples/ofd/` - 公开 OFD 样例
- 来源：OFD 联盟公开样例、脱敏政务文档

## 风险与应对

### 风险 1: OFD 规范复杂度高
**概率**: 高  
**影响**: 高  
**应对**: 分层实施，每层独立验收；优先覆盖常见场景

### 风险 2: 字体版权和嵌入
**概率**: 中  
**影响**: 中  
**应对**: 使用开源字体 fallback；提供字体缺失警告

### 风险 3: 性能问题（大文件）
**概率**: 中  
**影响**: 中  
**应对**: Worker 并行处理；动态分块转换

### 风险 4: 浏览器兼容性
**概率**: 低  
**影响**: 中  
**应对**: 使用标准 Web API；提供降级方案
