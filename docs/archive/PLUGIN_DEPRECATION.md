# 插件机制废弃说明

**版本**: v1.0.0  
**日期**: 2026-06-23  
**状态**: 已归档

---

## 决策

Trans2Former **不使用插件机制**。所有增强能力（OCR、模型缓存、格式转换）直接内置于核心模块。

## 背景

在早期版本（v0.1.0 - v0.5.0），项目曾考虑过插件架构来支持扩展能力：
- 用户可选安装 OFD、OCR、高保真渲染等插件
- 插件通过 `.t2f-plugin.json` 声明和加载
- 插件可以独立更新和分发

## 废弃原因

### 1. 安全风险
- 插件下载和安装增加攻击面
- 难以验证插件来源和完整性
- 违背"本地优先、零上传"的核心原则

### 2. 用户体验
- 插件安装流程复杂，增加使用门槛
- 插件版本不一致导致兼容性问题
- 用户难以判断哪些插件必需

### 3. 维护成本
- 插件 API 稳定性难以保证
- 插件与核心版本耦合严重
- 测试矩阵爆炸（核心 × 插件组合）

### 4. 资源效率
- 插件分发增加存储和带宽成本
- 插件加载增加启动时间
- 代码复用率低

## 替代方案

### 核心本地模块化设计

**原则**：
- 所有能力内置于核心模块
- 模型资源通过 vendor 脚本管理（SHA-256 校验）
- 随发布包分发，开箱即用

**实现**：
- 基础包：内置热门轻量格式（Markdown, HTML, JSON, CSV 等）
- 核心模块：OFD、OCR、版面分析、表格恢复等（代码内置，模型资源按需下载到本地 model-cache）
- 模型资源：不入 git，由 vendor 脚本下载并校验

**优点**：
- ✅ 安全可控：统一代码审核和签名
- ✅ 用户友好：一次安装，所有功能可用
- ✅ 易于维护：统一版本管理和测试
- ✅ 高效分发：模型资源按需下载，核心包保持轻量

## 历史版本

- **v0.6.0** (2026-05): 正式废弃插件机制，所有能力改为核心本地模块
- **v0.5.0** (2026-04): 提出插件路线调整计划
- **v0.4.0** (2026-03): 插件系统原型设计
- **v0.1.0** (2026-01): 建立插件准入规则

## 迁移指南

对于已有插件相关代码或文档：

1. **代码**：删除所有 PluginManager、Plugin API 相关代码
2. **文档**：更新为"核心本地模块"表述
3. **配置**：删除 `.t2f-plugin.json` 和 `plugin-patches`
4. **测试**：删除插件加载和兼容性测试
